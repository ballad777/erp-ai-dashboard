# 聚焦流程台工作區重新設計規格

日期：2026-06-15  
產品：智能金融資料分析  
狀態：已確認；實作優先順序受 `docs/audits/2026-06-15-project-plan-consistency-audit.md` 約束

## 1. 設計決策

採用視覺方向 A「聚焦流程台」。

核心原則是每個畫面只強調一個主要任務，先呈現使用者目前需要做的決定，再逐步展開模型、圖表、參數與分析結果。介面保留現有分析能力，但不再同時平鋪所有設定與資料。

參考影片的設計語言轉換為：

- 前景：目前可操作的分析步驟與主要按鈕。
- 中景：浮動工作區、系統建議與必要資料摘要。
- 背景：低對比網格、斜向柔和光帶與緩慢環境動態。
- 動態：內容替換、步驟推進、側邊建議滑入與結果分段揭露。

不複製參考作品的品牌、版面或顏色，只採用其空間層次與漸進式互動原則。

## 2. 目前問題

### 2.1 可讀性

- 部分次要文字受到舊 CSS 規則、透明度或主題變數影響，在淺色背景上接近白色。
- `text-muted-foreground` 被大量共用，無法保證所有舊元件在十種主題下都有足夠對比。
- 指標標籤、分析模式說明與模型描述沒有穩定的語意色彩邊界。

### 2.2 資訊層級

- 目標欄位、分析模式、模型策略、模型清單、AutoML 與圖表設定同時出現。
- 大型統計卡占用過多垂直空間，但沒有協助使用者做出當前決定。
- 使用者無法快速判斷「現在在哪一步」、「系統建議什麼」與「下一步要做什麼」。

### 2.3 互動

- 分析結果一次出現，缺少可理解的處理節奏。
- 配色選擇需要額外按下「套用」，與使用者期待不符。
- 手動模型數量多，缺少搜尋、分類與漸進展開。
- 選取、載入、完成與錯誤狀態的視覺回饋不夠集中。

## 3. 目標與非目標

本規格的介面改造只執行會改善可讀性、資訊層級、真實進度與操作效率的部分。依一致性稽核結論，不新增裝飾性主題、額外模型或與可信分析無關的動畫；安全、部署誠實性與可追溯基礎優先於視覺擴張。

### 3.1 目標

- 所有文字在十種主題下保持清楚可讀。
- 使用者進入模型分析後，五秒內理解目前步驟與主要操作。
- 自動分析成為預設且最明顯的路徑，手動控制仍完整保留。
- 配色點擊後立即套用並保存；前端即時圖表同步色彩，既有後端 PNG 不重新產生。
- 分析執行期間持續呈現真實後端進度，不使用假進度。
- 結果先顯示結論與下一步，再提供完整表格、圖表與程式碼。
- 桌面、平板與手機維持一致流程與可操作性。

### 3.2 非目標

- 不修改模型推薦演算法與後端評估公式。
- 不新增使用者未要求的分析模型或產品功能。
- 不重新設計首頁資訊架構。
- 不建立新的登入、帳號或雲端同步功能。
- 不以動畫取代真實 API 狀態。

## 4. 工作區資訊架構

模型分析工作區分成四個主要步驟：

1. 資料理解
2. 選擇方法
3. 執行分析
4. 整理結果

頂部固定顯示步驟列。已完成、目前與尚未開始的狀態必須有文字、圖示與顏色三種辨識方式，不能只靠顏色。

主內容採雙欄結構：

- 左側主要舞台：目前任務、必要設定、主要操作。
- 右側情境面板：系統建議、推薦理由、資料限制與預估執行內容。

資料列、特徵欄位與建議模型等摘要改為緊湊水平資訊列，不再使用三張大型卡片。

## 5. 選擇方法畫面

### 5.1 頁首

顯示：

- 目前步驟編號。
- 標題「選擇分析方式」。
- 一段不超過兩行的說明。
- 右上角保留語言與配色入口。

### 5.2 分析模式

提供三個互斥選項：

- 自動選擇
- 迴歸分析
- 分類分析

「自動選擇」預設選取，面積與說明略高於其他選項。每個選項必須包含：

- 模式名稱。
- 適用情境。
- 選取狀態。
- 鍵盤可操作的原生 radio 語意。

文字使用實色，不以 `opacity` 降低可見度。

### 5.3 系統建議

右側面板直接使用後端資料摘要與現有推薦資訊，說明：

- 為何判斷為迴歸或分類。
- 資料量是否需要交叉驗證。
- 缺失值、欄位型態或高相關特徵對分析的影響。
- 自動模式預計比較多少模型。

沒有足夠資訊時顯示明確空狀態，不生成假理由。

### 5.4 下一步

主要操作區固定放在目前步驟底部，內容包含：

- 標籤「清楚的下一步」。
- 下一個必要操作，例如「確認目標欄位後開始比較模型」。
- 單一主要按鈕。

按鈕只有在必要條件完成時可執行；停用時要顯示原因。

## 6. 手動模型與進階設定

手動設定不與自動路徑同時完全展開。

當使用者切換到手動模型：

- 以 180–240ms 高度與透明度轉場展開選擇區。
- 提供回歸、分類的分組篩選。
- 提供模型名稱搜尋。
- 模型選項使用高對比標題與說明。
- 顯示已選數量與清除選擇操作。

AutoML、圖表產出與其他進階選項維持可收合區塊。展開狀態需保存在目前資料集的工作區狀態中。

## 7. 分析執行體驗

分析仍使用現有真實 API 與工作進度資料。

執行期間顯示：

1. 判斷問題類型與資料規模
2. 準備特徵與資料分割
3. 訓練候選模型
4. 計算評估指標
5. 生成圖表與輸出

進度元件規則：

- 只顯示後端已回報或可由真實請求狀態確定的步驟。
- 不以固定計時器假裝分析進度。
- 目前步驟有柔和脈衝或掃描光帶，週期不得低於 1.5 秒。
- 可取消的工作保留取消按鈕。
- API 失敗時停在失敗步驟並顯示可重試操作。
- 支援 `prefers-reduced-motion`，停用位移與循環動畫。

## 8. 結果資訊層級

結果依序揭露：

1. 最佳模型與主要結論
2. 關鍵評估指標
3. 系統建議的下一步
4. 模型比較圖
5. 其他圖表
6. 完整模型表格
7. 產生的 Python 程式碼與 Notebook

結果不再一次顯示所有內容。第一個視窗高度內必須能看見最佳模型、核心指標與下一步。

程式碼使用內嵌檢視器顯示，下載是次要操作。程式碼檢視器支援：

- 語法高亮。
- 複製。
- 行號。
- Python 與 Notebook 分頁。
- 鍵盤操作。

## 9. 配色系統

保留目前十種配色。

互動改為：

- 點擊配色選項後立即套用。
- 立即寫入 `localStorage`。
- 選取後關閉選單。
- 移除「套用」與「取消」按鈕。
- Escape 或點擊外部只關閉選單，不還原配色。
- 鍵盤方向鍵移動焦點，Enter 或 Space 才正式選取。

主題切換只改變設計 tokens，不改變版面尺寸。

每個主題都要通過：

- 主要文字與背景至少 4.5:1。
- 次要文字與背景至少 4.5:1。
- 大型文字至少 3:1。
- 元件邊界、焦點環與互動狀態至少 3:1。

不再依賴通用 `text-muted-foreground` 處理關鍵說明文字。新增明確語意類別：

- `ui-copy-primary`
- `ui-copy-secondary`
- `ui-copy-tertiary`
- `ui-metric-label`
- `ui-option-description`

這些類別在所有主題中都必須使用不透明實色。

## 10. 動態與空間層次

### 10.1 背景

- 使用低透明度網格、柔和斜向光帶與局部色彩霧面。
- 動畫週期 18–28 秒。
- 僅使用 `transform` 與 `opacity`。
- 不讓光帶穿過主要文字區。
- 低效能與 reduced motion 裝置停用持續動畫。

### 10.2 工作區

- 工作區保持明確邊界與柔和陰影，形成中景。
- 目前步驟與主要按鈕形成前景。
- 側邊建議面板在步驟變更時以 300–420ms 滑入與淡入。
- 內容轉場可被新操作中斷，不鎖住輸入。

### 10.3 微互動

- hover：邊框與陰影輕微變化，位移最多 2px。
- press：縮小最多 0.99，持續 100–140ms。
- focus-visible：2px 焦點環與 2px 間距。
- success：狀態圖示與文字淡入，不使用強烈爆發動畫。
- error：錯誤區塊輕微水平位移一次，不循環。

## 11. 響應式設計

### 桌面

- 左側工具列、中央主要舞台與右側建議面板同時顯示。
- 主要內容最大寬度限制，避免超寬螢幕拉散資訊。

### 平板

- 左側工具列維持圖示模式。
- 右側建議面板移到主要內容下方。
- 「自動選擇」占滿第一列，迴歸與分類在第二列並排。

### 手機

- 工具列改為現有產品殼層的頂部精簡導覽。
- 步驟列可水平捲動，但目前步驟必須自動進入視野。
- 分析模式單欄排列。
- 主要操作可固定在安全區上方。
- 所有觸控目標至少 44×44px。

## 12. 元件與程式邊界

預計拆分以下可重用元件：

- `AnalysisStepRail`：步驟狀態與導覽。
- `AnalysisFocusStage`：目前步驟標題、說明與主要操作。
- `AnalysisModeSelector`：自動、迴歸與分類選擇。
- `DatasetMetricStrip`：緊湊資料摘要。
- `AnalysisRecommendationPanel`：真實推薦理由。
- `ModelSelectionDrawer`：手動模型搜尋、分類與選取。
- `AnalysisProgressStage`：真實工作進度、取消與失敗狀態。
- `ResultPrioritySummary`：最佳模型、核心指標與下一步。
- `InlineCodeViewer`：程式碼內嵌檢視。

`ModelAnalysisPanel` 保留資料取得與流程協調責任，不再直接承擔所有視覺與選擇元件。

`ThemeProvider` 對外改為單一 `selectTheme(themeId)` 操作，負責套用與保存。`ThemePicker` 只負責顯示與選取。

## 13. 資料流與狀態

1. 資料集由現有 `WorkspaceProvider` 提供。
2. 目標欄位、分析模式、模型選擇與圖表選擇依 `workspaceSourceKey` 保存。
3. 自動模式以現有 API 請求 `auto`，後端仍負責最終模型推薦。
4. 右側建議只使用資料集分析回傳與模型 API 回傳，不建立前端假結論。
5. 執行結果寫回目前資料集的工作區狀態，返回首頁再回到工作區時不得消失。
6. 主題選取寫入獨立的產品主題儲存鍵。

## 14. 錯誤與邊界狀態

- 無目標欄位：停用執行並說明如何修正。
- 手動模式未選模型：顯示行內錯誤並將焦點移至模型區。
- 手動圖表未選任何項目：顯示行內錯誤。
- 合併分析不足兩個成功檔案：保留現有阻擋邏輯。
- API 逾時或中斷：保留已完成步驟，提供重試。
- 部分模型失敗：結果頁列出成功模型，失敗模型放入次要詳情。
- 主題儲存失敗：目前頁面仍立即套用，僅不持久化。
- 暗色主題圖表與程式碼區必須使用對應高對比 tokens。

## 15. 驗收與測試

### 自動測試

- 主題點擊後立即套用並保存。
- ThemePicker 不再出現套用與取消按鈕。
- 分析模式可用鍵盤切換。
- 自動與手動模式只顯示各自必要內容。
- 無效設定阻止 API 呼叫。
- 後端進度正確映射到畫面步驟。
- 返回首頁再回到工作區後，資料集與分析結果仍保留。
- 中文與英文文案均可完整顯示。

### 視覺與互動驗證

- 桌面：1440×900、1280×800。
- 平板：1024×768、768×1024。
- 手機：390×844、375×667。
- 瀏覽器縮放：80%、100%、125%、150%。
- 十種主題逐一檢查文字對比。
- 淺色與深色主題檢查圖表、表格、程式碼與錯誤狀態。
- 鍵盤完整走完選擇、執行、查看結果與配色切換。
- reduced motion 下不應有持續背景與位移動畫。

### 完成條件

- production build 通過。
- 前端測試通過。
- 後端既有分析測試通過。
- Browser 實際執行一次 CSV 上傳、自動模型分析、結果查看與程式碼預覽。
- Browser 實際驗證手機版無遮擋、橫向溢出與文字不可讀問題。

## 16. 實作範圍順序

1. 修正語意文字 tokens 與十種主題對比。
2. 改為配色點擊即套用。
3. 拆分模型分析元件。
4. 建立聚焦流程台與右側建議面板。
5. 建立手動模型漸進展開。
6. 重做真實分析進度與分段結果。
7. 加入程式碼內嵌檢視。
8. 完成響應式、動態、accessibility 與 production 驗證。
