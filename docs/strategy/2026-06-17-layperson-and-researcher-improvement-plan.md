# 讓一般人讀懂、讓研究者深挖的完整改善計畫

日期：2026-06-17  
專案：智能金融資料分析  
目標：同一套系統同時服務「沒有資料科學背景的一般使用者」與「想深度研究資料、模型、金融指標的人」。

## 1. 核心結論

目前系統已經能完成真實資料上傳、資料摘要、模型分析、金融指標、圖表、報告與部分白話摘要；但目前「看得懂」主要集中在報告頁，資料頁、模型頁、金融頁仍有大量技術名詞與指標直接暴露。

下一階段不應只加 tooltip，也不應把產品拆成兩套。正確方向是建立同一個分析結果的三層閱讀架構：

1. **一般人層**：一句話結論、發生什麼、代表什麼、可信度如何、下一步做什麼。
2. **決策證據層**：系統為什麼這樣判斷、哪些資料或圖表支持、哪些地方要小心。
3. **研究者層**：公式、指標、模型設定、資料處理流程、驗證方法、原始輸出與可重現資產。

產品定位應從「會產生分析結果的工具」升級成「會教使用者正確閱讀分析結果的雙層分析工作台」。

## 2. 目標使用者

### 2.1 一般使用者

代表族群：

- 老闆、主管、業務、財務、行銷、營運。
- 剛接觸資料分析的學生。
- 只想知道資料代表什麼、風險在哪裡、下一步怎麼做的人。

他們需要：

- 不需要懂 RMSE、RSI、MACD、VaR 也能理解結論。
- 每個數字都要有「白話翻譯」。
- 不被模型或圖表嚇到。
- 明確知道哪些結論可信，哪些只是參考。
- 能把結果帶去開會或寫作業。

### 2.2 深度研究者

代表族群：

- 資料科學學生、分析師、量化研究者、工程師。
- 想檢查模型是否合理的人。
- 想重現分析、比較方法、改模型的人。

他們需要：

- 原始指標、公式、參數、模型方法。
- 資料清理規則與欄位處理流程。
- 訓練/測試切分、交叉驗證、回測。
- 圖表背後資料、下載檔、Notebook、程式碼。
- 模型限制、偏誤、風險與可重現性。

1. 白話解釋太晚出現。  
   部分白話能力已前移，但仍需要逐頁驗收是否真的出現在資料、模型、金融的第一屏，而不是只存在於 schema 或元件。

2. 指標沒有「讀法」。  
   指標字典已存在，但仍要驗證所有主要 UI 都有接上 `terms` 與 `MetricExplainer`，並確認每個指標都有「這次代表什麼」。

3. 預測沒有可信度防護。  
   `forecast_reliability` 已存在；下一步要驗證前端警示是否夠明顯，且文案是否能阻止一般人把基準情境估計誤讀為投資預測。

4. 「價格」用語不夠泛化。  
   部分後端與前端已改成「價格/指數/數值」，但仍要全站掃描舊文案與圖表標籤。

5. 沒有明確的閱讀模式。  
   閱讀深度控制已存在；缺口改為：它是否在所有主要頁面都容易被找到，且不同深度是否真的改變資訊量。

6. 研究者需要的驗證資訊不足。  
   `research_details` 已存在；但交叉驗證、時間序列回測、圖表原始資料下載、完整模型參數與重現包仍需要補強。

7. 缺少使用者理解度驗收。  
   目前測試多偏功能正確性，還沒有「一般人是否看懂」的產品驗收標準。

8. 文件與實作狀態需要持續同步。  
   原計畫中的部分項目已經被實作，後續 roadmap 必須明確標記已完成、部分完成與未完成，避免重複開發。

## 4. 產品設計原則

### 4.1 不做兩套產品

不要做「一般版」和「專家版」兩個互相割裂的產品。所有分析結果都應共享同一份資料與同一套後端輸出，只是前端逐層揭露。

建議層級：

- 預設：簡明模式。
- 使用者展開：看證據。
- 使用者進一步切換：研究模式。

### 4.2 每個技術結果都要有四件事

每個模型、指標、圖表、預測都必須回答：

1. **這是什麼？**
2. **這次數字代表什麼？**
3. **我該相信到什麼程度？**
4. **下一步要做什麼？**

### 4.3 預設降低誤解，不預設提高自信

金融與模型預測最容易被一般人誤解。系統應主動標示：

- 這不是投資建議。
- 這不是保證未來會發生。
- 這次模型可信度高/中/低。
- 哪些資料品質問題會影響結論。

### 4.4 深度研究不能藏起來

簡明模式不等於刪掉細節。研究者應該能一鍵展開：

- 原始資料摘要。
- 方法說明。
- 模型參數。
- 指標公式。
- 圖表資料。
- 可重現程式碼與 Notebook。

## 5. 建議資訊架構

每個分析結果頁統一採用四層結構。

### 5.1 第一層：一句話結論

畫面上方固定顯示：

- 目前最重要結論。
- 可信度標籤。
- 主要風險。
- 建議下一步。

範例：

> 這份 NASDAQ 指數資料顯示近期短線偏弱，但預測模型可信度偏低，因為線性預測與最新指數落差過大。建議先看趨勢與風險，不要把預測點當成投資判斷。

### 5.2 第二層：一般人摘要

固定四張卡：

- 發生了什麼
- 為什麼重要
- 要小心什麼
- 下一步做什麼

### 5.3 第三層：證據與圖表

每張圖表下方固定顯示：

- 這張圖在看什麼。
- 這次圖表最重要的變化。
- 這張圖不能證明什麼。
- 建議下一步。

### 5.4 第四層：研究細節

使用 details / drawer / tabs 展開：

- 欄位偵測規則。
- 資料清理規則。
- 指標公式。
- 模型訓練設定。
- 交叉驗證或回測結果。
- 原始 JSON、CSV、圖表檔、程式碼、Notebook。

## 6. 後端改善計畫

### 6.1 建立通用解釋資料結構

新增或擴充後端 schema，讓資料、模型、金融、報告都能回傳一致的解釋欄位。

建議結構：

```json
{
  "plain_summary": {
    "headline": "一句話結論",
    "what_happened": "發生了什麼",
    "why_it_matters": "為什麼重要",
    "risk": "要小心什麼",
    "next_action": "下一步"
  },
  "confidence": {
    "level": "high | medium | low",
    "reason": "可信度原因",
    "blocking_issues": ["影響可信度的問題"]
  },
  "evidence": [
    {
      "label": "支持此結論的證據",
      "value": "數值或觀察",
      "source": "來自哪個欄位、圖表或模型"
    }
  ],
  "terms": [
    {
      "term": "RSI",
      "plain_explanation": "用來看短期是否偏熱或偏弱的指標",
      "technical_definition": "Relative Strength Index",
      "how_to_read": "通常低於 30 偏弱，高於 70 偏熱",
      "caveat": "不能單獨作為買賣依據"
    }
  ],
  "research_details": {
    "method": "方法",
    "assumptions": ["假設"],
    "parameters": {},
    "limitations": ["限制"],
    "artifacts": []
  }
}
```

目前 `insight_narrative.py` 已有決策摘要能力。建議把它從「報告頁專用」升級成「全系統解釋引擎」。

### 6.3 建立指標字典

新增 `backend/app/services/metric_glossary.py` 或等價模組。

每個指標包含：

- 顯示名稱。
- 白話說明。
- 技術定義。
- 公式。
- 常見判讀區間。
- 何時容易誤判。
- 適用/不適用情境。

第一批指標：

- 資料完整度
- 缺失值
- 平均數 / 中位數 / 標準差
- RMSE
- MAE
- R2
- Accuracy
- F1-score
- Feature importance
- Residual
- Return
- Volatility
- Moving Average
- RSI
- MACD
- VaR
- Forecast

### 6.4 金融預測可信度檢查

新增預測防護邏輯：

- 若預測值與最新值差距超過門檻，標示低可信度。
- 若模型只是線性基準，標示「基準預測，不適合直接當市場預測」。
- 若資料跨越數十年且趨勢非線性，提醒應使用近年窗口或回測。
- 若缺值集中在交易日或資料異常，降低可信度。

建議回傳：

```json
{
  "forecast_reliability": {
    "level": "low",
    "reason": "預測值與最新值落差過大，且目前使用簡化線性趨勢模型。",
    "recommended_action": "改用近 1 年或近 5 年資料視窗，並加入回測誤差後再解讀。",
    "should_show_warning": true
  }
}
```

每次分析都應回傳或可下載：

- run id
- dataset fingerprint
- 使用欄位
- 排除欄位
- 缺失值處理
- 類別欄位編碼
- train/test split
- random seed
- 模型清單
- 超參數
- 指標
- 圖表原始資料
- 生成時間
- 程式碼版本

這些資料是研究者信任系統的基礎。

## 7. 前端改善計畫

### 7.1 全域模式切換

新增「閱讀深度」控制：

- `簡明`
- `標準`
- `研究`

建議預設：

- 新使用者：簡明。
- 已展開研究區或手動設定後：記住偏好。

不要把簡明模式做成玩具版；簡明模式仍要保留風險和限制，只是不用公式壓迫使用者。

### 7.2 新增共用元件

建議建立：

- `InsightHeadline`
- `PlainSummaryGrid`
- `ConfidenceBadge`
- `MetricExplainer`
- `ChartStoryPanel`
- `TermDefinitionPopover`
- `ResearchDetailsDrawer`
- `MethodCard`
- `ForecastReliabilityNotice`
- `EvidenceList`

這些元件要跨資料、模型、金融、報告頁共用。

### 7.3 資料頁改善

目前資料頁顯示資料列、欄位、缺失值、數值摘要。改善後要多回答：

- 這份資料大概是什麼類型。
- 它適合做什麼分析。
- 目前最大資料品質問題是什麼。
- 缺失值是否可能是正常現象。
- 哪些欄位可能是時間、價格、目標欄位、類別欄位、ID 欄位。

建議新增區塊：

- 「這份資料可以回答什麼？」
- 「正式分析前先檢查這些」
- 「欄位角色推測」
- 「一般人版資料品質說明」
- 「研究者版欄位型別與缺失矩陣」

### 7.4 模型頁改善

模型頁應把技術設定轉成自然問題：

目前：

- 選目標欄位
- 選分析模式
- 選模型
- 選圖表

改善後：

- 你想讓系統預測什麼？
- 這個目標比較像數字預測還是分類判斷？
- 系統為什麼推薦這些模型？
- 最佳模型比 baseline 好多少？
- 錯誤主要出現在哪些資料？
- 這個模型可不可以拿來決策？

每個模型結果都要有：

- 白話說明。
- 模型用途。
- 指標讀法。
- 和 baseline 比較。
- 可信度。
- 何時不該相信。
- 研究細節。

### 7.5 金融頁改善

金融頁是最容易造成誤解的地方，必須優先改善。

新增第一屏：

- 一句話趨勢。
- 風險等級。
- 預測可信度。
- 非投資建議。
- 下一步。

KPI 不要只顯示：

- 最新價格
- 最新報酬率
- 波動率
- RSI
- VaR

要改成「數值 + 白話解讀」：

- 最新報酬率 `-1.15%`：最近一期下跌。
- 波動率 `4.96%`：近期變動幅度偏高/中/低。
- RSI `46.76`：接近中性，不是明顯過熱或過弱。
- VaR 95% `2.04%`：以歷史資料估計，正常極端情況下可能有約 2% 等級的單期下行風險。

圖表要新增：

- 「這張圖該看哪裡」
- 「這次最重要變化」
- 「可能誤解」
- 「研究者細節」

預測圖必須新增警告：

- 若預測模型是簡化模型，明確標示。
- 若預測落差過大，使用紅色/警示卡。
- 若沒有回測，不要用「預測」語氣，改成「基準情境估計」。

### 7.6 報告頁改善

報告頁應輸出兩種報告：

1. **一般人報告**
   - 一頁式摘要。
   - 發生什麼。
   - 為什麼重要。
   - 風險。
   - 建議下一步。
   - 不需要公式。

2. **研究附錄**
   - 資料來源。
   - 欄位表。
   - 指標公式。
   - 模型設定。
   - 驗證方法。
   - 圖表原始資料。
   - 限制與假設。

Word / PDF / Markdown 都應採這種雙層結構。

## 8. 使用體驗流程

### 8.1 一般人路徑

1. 上傳資料。
2. 系統顯示「這份資料看起來像什麼」。
3. 系統說明「能做哪些分析」。
4. 使用者點選「用白話分析」。
5. 系統先顯示一句話結論。
6. 使用者看到風險與下一步。
7. 若需要，展開圖表說明。
8. 匯出一般人報告。

### 8.2 研究者路徑

1. 上傳資料。
2. 檢查欄位型別與缺失值。
3. 設定目標欄位、模型、圖表。
4. 查看 baseline、模型比較與驗證結果。
5. 展開研究細節。
6. 下載原始圖表資料、Notebook、程式碼。
7. 重跑或調整模型。
8. 匯出研究附錄。

## 9. 分階段執行計畫

### Phase 1：建立可讀性基礎

目標：讓所有頁面開始有一致的白話摘要與指標說明。

目前狀態：**部分完成**。指標字典、schema、共用解釋元件與部分頁面接入已存在。

工作：

- 驗證資料、模型、金融頁是否都顯示 `InsightHeadline`。
- 驗證每個主要回應都有 `plain_summary`、`confidence`、`evidence`、`terms`。
- 補齊尚未接入 `MetricExplainer` 或 `PlainSummaryGrid` 的頁面。
- 全站掃描舊文案，把不精準的「價格欄位」改為「價格/指數/數值欄位」。
- 補齊所有高風險結果的「限制」與「不構成投資建議」。

驗收：

- 一般人看資料頁第一屏，能知道這份資料大概能做什麼。
- 一般人看金融頁，不需要懂 RSI/MACD/VaR 也能知道目前趨勢與風險。
- 每個主要指標都有白話說明。
- `MetricExplainer` 在 simple 模式最多顯示核心指標，在 research 模式顯示完整公式與技術定義。
- 沒有任何主要分析結果只顯示數值而沒有解釋。

### Phase 2：建立可信度與預測防護

目標：避免使用者過度相信模型與金融預測。

目前狀態：**部分完成**。`confidence` 與 `forecast_reliability` 已進入 schema 與部分 UI。

工作：

- 驗證金融預測偏離最新值超過門檻時，前端一定顯示高可見度警告。
- 驗證模型結果的 `confidence` 是否同時考慮資料品質、baseline 差異與模型指標。
- 補齊資料品質對模型可信度的可視化說明。
- 對時間序列補入更明確的「未回測」狀態。
- 增加測試案例：預測值明顯不合理時，API 和 UI 都必須標示低可信度。

驗收：

- 當預測明顯不合理時，系統不會只顯示預測值，會明確說明為何不能直接相信。
- 使用者能看出「資料問題」如何影響「模型可信度」。
- 金融頁不可把簡化線性外推稱為確定預測，應顯示為基準情境估計。
- 低可信度結果在 simple 模式也必須看得到，不可只藏在研究模式。

### Phase 3：建置研究模式

目標：讓深度研究者能追溯每個結果。

目前狀態：**已建立骨架，仍需補強內容**。`ResearchDetailsDrawer` 與 `research_details` 已存在。

工作：

- 驗證研究模式切換是否在資料、模型、金融、報告頁一致運作。
- 補齊方法、參數、假設、限制。
- 模型頁顯示 train/test split、random seed、候選模型、超參數。
- 金融頁顯示公式、窗口大小、回測結果。
- 圖表可下載原始資料。
- 建立 run-level reproducibility bundle，包含資料指紋、schema、模型設定、圖表資料、生成程式碼與 Notebook。

驗收：

- 研究者能回答「這個結果怎麼算出來」。
- 研究者能用下載資產重現主要結果。
- 研究者能看出哪些結果尚未完成交叉驗證或回測。

### Phase 4：報告輸出雙層化

目標：讓報告同時適合一般讀者和研究者。

工作：

- 一般人摘要報告。
- 研究附錄。
- 每張圖表附白話解讀與技術說明。
- 報告加入資料來源、分析時間、run id、限制。
- 匯出 Markdown / Word / PDF 結構一致。

驗收：

- 一般人可以只看前兩頁。
- 研究者可以看附錄重現分析邏輯。

### Phase 5：理解度與信任測試

目標：驗證產品真的能讓非資料背景使用者看懂。

工作：

- 找 5 位非資料背景使用者測試。
- 找 3 位資料/金融背景使用者測試。
- 記錄使用者是否能回答：
  - 這份資料是什麼？
  - 系統最重要結論是什麼？
  - 哪些地方不能直接相信？
  - 下一步該做什麼？
  - 如果想深挖，要去哪裡看？
- 針對錯誤理解修正文案與 UI。

驗收指標：

- 非資料背景使用者 80% 能在 3 分鐘內說出主要結論。
- 非資料背景使用者 80% 能指出至少一個限制或風險。
- 研究者 80% 能找到模型設定與原始輸出。
- 高風險預測不被使用者誤讀為確定結論。

## 10. 優先順序

- 補齊資料/模型/金融頁第一屏白話摘要的缺漏。
- 全站掃描並移除不精準的「價格」殘留文案。
- 確認每張圖表都有 `chart_stories` 並在 UI 呈現「這張圖怎麼看」。
- 增加回歸測試，防止未來 API 回應缺少 `plain_summary`、`confidence`、`terms` 或 `research_details`。

### P1：高價值

## 11. 風險與取捨

### 11.1 太多文字會壓垮使用者

解法：預設只顯示一句話結論與三到四張重點卡，細節放在展開層。

### 11.2 白話可能過度簡化

解法：每個白話解釋都連到技術定義與公式，並保留限制。

### 11.3 LLM 摘要可能不穩定

解法：核心解釋先用規則式模板，LLM 只做語氣整理；未設定 API key 時不可假裝已使用 AI。

### 11.4 金融分析容易被當成投資建議

解法：所有金融頁與報告都要固定顯示非投資建議、模型限制與資料區間。

### 11.5 研究模式可能讓 UI 變複雜

解法：使用抽屜、details、tab，不把研究細節放在第一屏。

## 12. 建議成功指標

產品指標：

- Time to first understood insight：一般人理解第一個結論的時間。
- Plain-language summary read rate：白話摘要閱讀率。
- Research details open rate：研究細節展開率。
- Export completion rate：報告匯出完成率。
- Forecast warning comprehension：使用者是否理解預測限制。

品質指標：

- 每個主要指標都有 glossary entry。
- 每個圖表都有 chart story。
- 每個預測都有 reliability status。
- 每次分析都有 run metadata。
- 前端 desktop / mobile / keyboard / reduced-motion 驗收通過。

## 13. 最終產品形態

完成後，使用者看到的不是一堆模型指標，而是一個清楚的閱讀路徑：

1. **先看結論**：這份資料告訴你什麼。
2. **再看風險**：哪裡不能直接相信。
3. **再看證據**：哪些圖表與數字支持。
4. **最後深挖**：方法、公式、模型、資料、程式碼、Notebook。

這樣一般人能讀懂，研究者也不會覺得被簡化或被阻擋。

## 14. 建議下一步

先做 Phase 1 與 Phase 2 的**驗收與補強**。理由是這兩階段的主要能力已經部分落地，但還不能只因為 schema 和元件存在就判定產品完成。真正要確認的是一般人是否在畫面上看得到、讀得懂、且不會誤信預測。

建議第一個實作切點：

1. 寫一組 API 合約測試：資料、模型、金融回應都必須包含 `plain_summary`、`confidence`、`evidence`、`terms`、`research_details`。
2. 寫一組前端測試：simple 模式下仍看得到低可信度警告與下一步。
3. 用 FRED NASDAQCOM 這類真實資料跑一次金融分析，確認 `forecast_reliability` 會攔住不合理線性外推。
4. 全站掃描 UI 文案，移除「價格」殘留誤導，統一成「價格/指數/數值」或依資料型態動態命名。
5. 針對非資料背景使用者做 5 人理解度測試，測試問題固定化。

這是最小但影響最大的下一步，不需要先重構整個產品，也能避免重複開發已經存在的能力。
